# SpecKit 章程

## 核心原则

### I. 代码质量标准

所有代码**必须**遵循以下质量标准：

- **可读性优先**: 代码首先是写给人看的，其次才是让机器执行。清晰的命名、一致的格式和逻辑结构是强制性的。
- **DRY 原则**: 禁止重复。提取可复用的组件、函数和工具。
- **单一职责**: 每个模块、类和函数**必须**只有一个明确的目的。
- **类型安全**: 在语言支持的情况下，所有代码**必须**使用强类型。没有明确理由不得使用 `any` 类型。
- **文档**: 公共 API **必须**有文档说明。复杂逻辑**必须**包含解释"为什么"而非"是什么"的内联注释。
- **Lint 合规**: 所有代码**必须**通过配置的 lint 规则，无警告。

**原理**: 高质量的代码减少技术债务，加速新人上手，并减少 bug。可读的代码才是可维护的代码。

### II. 测试标准 (不可协商)

测试不是可选的。每个功能**必须**包含全面的测试覆盖：

- **测试优先开发**: 在实现之前编写测试。红-绿-重构周期是强制性的。
- **最低覆盖率**: 所有新代码**必须**达到最低 80% 的测试覆盖率；关键路径需要 100%。
- **测试金字塔**: 优先单元测试 (70%)，辅以集成测试 (20%) 和端到端测试 (10%)。
- **独立性**: 每个测试**必须**独立且幂等。测试之间不共享状态。
- **确定性**: 测试**必须**每次运行产生相同结果。不允许有 flaky 测试。
- **快速反馈**: 单元测试**必须**在 100ms 内完成。完整测试套件**必须**在 5 分钟内完成。

**原理**: 测试是支持自信重构和持续交付的安全网。未经测试的代码是迟早会出问题的代码。

### III. 用户体验一致性

所有面向用户的功能**必须**提供一致、直观的体验：

- **设计系统合规**: 所有 UI 组件**必须**使用已建立的设计系统（颜色、排版、间距、组件）。
- **响应式设计**: 所有界面**必须**在桌面、平板和移动设备上无缝工作。
- **无障碍性 (a11y)**: 所有功能**必须**符合 WCAG 2.1 AA 标准。键盘导航和屏幕阅读器支持是强制性的。
- **错误处理**: 错误消息**必须**对用户友好、可操作，并在语气和格式上保持一致。
- **加载状态**: 每个异步操作**必须**显示适当的加载、成功和错误状态。
- **一致的导航**: 导航模式**必须**在所有视图中可预测且一致。

**原理**: 一致的 UX 减少用户认知负荷，提高满意度，并建立对产品的信任。

### IV. 性能要求

性能是一个功能。所有代码**必须**达到定义的性能基准：

- **响应时间**: API 端点**必须**在 200ms (p95) 内响应。用户界面**必须**在 1 秒内渲染。
- **包大小**: 初始 JavaScript 包**必须**小于 200KB (gzip)。懒加载非关键代码。
- **内存使用**: 无内存泄漏。峰值内存使用**必须**保持在基线的 150% 以内。
- **数据库查询**: 禁止 N+1 查询。所有查询**必须**经过优化和索引。
- **缓存策略**: 在所有层级（浏览器、CDN、应用、数据库）实现适当的缓存。
- **资源优化**: 图片**必须**经过优化。未使用的代码**必须**被消除。

**原理**: 性能直接影响用户满意度、转化率和 SEO 排名。缓慢的应用程序会失去用户。

## 开发工作流

### 代码审查要求

- 所有代码变更在合并前**必须**由至少一名其他开发人员审查。
- 审查者**必须**验证是否符合本章程。
- 自动化检查（lint、test、type-check）**必须**在人工审查前通过。
- 复杂变更需要架构审批。

### 提交标准

- 提交**必须**遵循约定式提交格式: `type(scope): description`。
- 每次提交**必须**是原子的，代表一个单一的逻辑变更。
- 提交消息在适用时**必须**引用相关的 issue/工单编号。

### 分支策略

- 主分支**必须**始终可部署。
- 功能分支**必须**是短生命周期的（在 1 周内合并）。
- 所有合并到主分支的操作**必须**使用 squash merge 并保持干净的提交历史。

## 质量门禁

在代码合并到主分支之前，**必须**通过以下检查：

1. **自动化测试**: 所有测试**必须**通过（单元、集成、端到端，如适用）。
2. **静态分析**: Linting 和类型检查**必须**零错误通过。
3. **代码覆盖率**: 覆盖率**不得**下降；新代码**必须**达到最低阈值。
4. **性能预算**: 包大小和性能指标**必须**保持在预算范围内。
5. **无障碍性审计**: 不得引入新的 a11y 违规。
6. **人工审查**: 至少一名团队成员的批准审查。

## 治理

本章程优先于所有其他开发实践。期望所有团队成员理解并遵循这些原则。

### 修订流程

1. 提议的修订**必须**附有原理说明。
2. 修订需要至少 2 名高级团队成员的批准。
3. 对原则的破坏性变更需要团队范围的讨论。
4. 所有修订**必须**更新版本号和最后修订日期。

### 合规审查

- 每季度对代码库进行章程合规性审查。
- 违规行为**必须**被跟踪和修复。
- 重复违规会触发流程改进讨论。
